# Patch Lookup Patterns

How to find authoritative remediation data for a CVE without making the user do the work.

## The hierarchy of sources

1. **Securin vulnerability record** (`searchVulnerabilityData`) — already-normalized fields: `remediation`, `fixedInVersions[]`, `vendorAdvisoryUrl`, `patchReferences[]`. Try this first.
2. **Vendor advisory** — the primary source of truth for upgrade steps. Find it, read it.
3. **KB article** (vendor knowledge base) — often deeper detail on edge cases, prerequisites, known issues.
4. **CISA KEV entry** — for KEV CVEs, CISA's catalog page summarizes the mitigation.
5. **NVD entry** — last resort; cross-references third-party advisories but is often sparse on remediation detail.

## Finding the vendor advisory

Web search query templates:

```
site:<vendor-domain> CVE-XXXX-YYYY
CVE-XXXX-YYYY security advisory <vendor>
"CVE-XXXX-YYYY" fixed in   # narrows to docs that state a fixed version
```

Known vendor advisory URL patterns (shortcut when vendor is clear):

| Vendor | Pattern |
|---|---|
| Microsoft | `msrc.microsoft.com/update-guide/vulnerability/CVE-XXXX-YYYY` |
| Palo Alto | `security.paloaltonetworks.com/CVE-XXXX-YYYY` |
| Cisco | `sec.cloudapps.cisco.com/security/center/content/CiscoSecurityAdvisory/...` |
| Red Hat | `access.redhat.com/security/cve/CVE-XXXX-YYYY` |
| Ubuntu | `ubuntu.com/security/CVE-XXXX-YYYY` |
| GitHub (GHSA) | `github.com/advisories/GHSA-...` |
| Apache | `httpd.apache.org/security/vulnerabilities_*.html` (or project-specific) |

## Constructing the upgrade command

Tailor to the asset's platform. Examples:

### OS-level packages
```bash
# Debian / Ubuntu
sudo apt-get update && sudo apt-get install --only-upgrade <pkg>=<fixed-version>

# RHEL / CentOS / Fedora
sudo dnf upgrade <pkg>-<fixed-version>     # dnf on modern
sudo yum update <pkg>-<fixed-version>      # yum on legacy

# Alpine
apk upgrade <pkg>

# Windows
# Usually a KB patch — reference KB id from advisory
# `wusa.exe <kb>.msu /quiet /norestart` or via WSUS / WSUS-replacement
```

### Application / language packages
```bash
# Java / Maven
<update pom.xml / gradle.build to fixed-version, rebuild, redeploy>

# Node / npm
npm install <pkg>@<fixed-version>
npm audit fix                       # batch fix mode

# Python / pip
pip install --upgrade <pkg>==<fixed-version>

# Go
go get <module>@<fixed-version>

# .NET
dotnet add package <pkg> --version <fixed-version>
```

### Container / image
Pin to a fixed tag or digest; rebuild and redeploy.

### Network appliance
Usually a firmware upload + reboot. Reference vendor advisory's exact procedure.

## Workaround detection

If no patch is available yet, look for:
- Config flag / feature disable instructions.
- Firewall / IDS / WAF rule to block the exploit path.
- Kernel-level mitigation (SELinux, AppArmor policy).
- Account / permission restrictions.

These live in the advisory, often in a "Mitigations" or "Workarounds" section. Quote the exact steps.

## Compensating controls

When neither patch nor workaround is feasible, recommend detection + containment:

- **Detection:** Sigma / Yara / Snort / Suricata / Zeek rule for the CVE's exploit signature. Many are published to community repos (SigmaHQ on GitHub).
- **Network:** Egress/ingress firewall rules blocking the attack path.
- **Host:** EDR policy (block specific binaries / behaviors).
- **Identity:** MFA / conditional access for affected services.

Reference public detection rules rather than inventing them — cite the source.

## Pre-patch risk

Before recommending the patch, call:

```text
searchExposureData
filter: exposure.assetId = '<affected asset id>' AND exposure.status = 'Open'
```

List the other open exposures on the same asset. This surfaces:
- Other services on the host that might be affected by the patch (e.g., a kernel patch impacting a legacy driver).
- Recently-closed exposures on the same asset (recency of the last patch — tells you patching discipline).
- Dependency risk — if the host is a database, reboot planning matters.

## Known regressions

When reading the advisory and KB article, look for phrases like:
- "Known Issues"
- "Applies To"
- "Troubleshooting"

Surface any regressions. Include the mitigation (often a subsequent patch or config tweak).

## Handoff templates

### Jira ticket body

```
[SECURITY] Patch <CVE> on <asset hostname>

**Exposure:** <exposure-id>
**CVE:** <CVE> (KEV: yes/no, Severity: <>)
**Asset:** <hostname> (<criticality>, <workspace>)

## Required action
Upgrade <package> to <fixed-version>:
    <exact command>

## Prerequisites
<list>

## Vendor advisory
<url> — key excerpt:
> <quoted paragraph>

## Rollback
<steps>

## Deep links
- Exposure: <createDeepLink url>
- Asset: <createDeepLink url>
- Vulnerability: <createDeepLink url>

Generated by Claude + securin-remediation-guidance. Owner: <from exposure record>.
```

### ServiceNow ticket body

Same structure, adapt fields to your SN template. Include a `u_securin_exposure_id` custom field if your instance has it.

## Don't

- Don't tell the user to "check the vendor advisory" — read it yourself and quote the relevant steps.
- Don't invent an upgrade command you haven't verified against the advisory.
- Don't skip the pre-patch risk section for prod assets.
- Don't create tickets automatically (write-actions out of scope in M1).
